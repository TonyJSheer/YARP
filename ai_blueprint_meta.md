# AI Blueprint Meta — Architecture Defaults & Decision System

This is the engineering rulebook for AI-assisted projects built with this framework.

**Audience**: Human + architect agent during the planning phase.
Coding agents work from the resolved output (`PROJECT_BLUEPRINT.md` and `docs/AGENTS.md`) — they do not need to read this document.

---

## Document Model

The framework uses three documents in sequence:

1. **Project Specification** — filled-in `project_spec_template.md` — WHAT to build
2. **This document** — HOW to build it (architecture defaults and decision rules)
3. **Project Blueprint** — `PROJECT_BLUEPRINT.md`, generated by the architect agent — the concrete, resolved architecture for this specific project

The architect agent reads (1) + (2) and produces (3). Coding agents work from (3).

---

## Decision Slots

Architecture decisions are expressed as named slots. Each slot has a default that applies unless the project spec or human overrides it.

**These decisions are made by the human + architect agent during planning. The coding agent receives them as resolved requirements — it does not make or debate these decisions.**

| Slot | Description | Default | Options |
|---|---|---|---|
| `api_style` | How clients communicate with the backend | `rest` | `rest`, `graphql`, `redis_event_layer`, `hybrid` |
| `auth_method` | User authentication mechanism | `token` | `token`, `oauth`, `magic_link`, `session` |
| `mobile_support` | Mobile app requirement | `phase_2` | `none`, `phase_2`, `required_at_launch` |
| `redis_hosting` | How Redis is hosted | `self_hosted_ecs` | `self_hosted_ecs`, `elasticache` |
| `realtime` | Real-time push to clients | `none` | `none`, `sse`, `websockets`, `redis_pubsub` |
| `multitenancy` | Multi-organisation support | `no` | `no`, `yes` |
| `background_jobs` | Background processing mechanism | `redis_workers` | `none`, `redis_workers`, `lambda`, `celery` |
| `file_storage` | User file/upload handling | `s3` | `none`, `local`, `s3` |
| `deployment_target` | Hosting platform | `ecs_fargate` | `ecs_fargate`, `ec2`, `lambda_only`, `fly_io`, `railway` |
| `database_migrations` | Schema change management | `alembic` | `alembic`, `manual` |
| `admin_interface` | Admin operations surface | `protected_endpoints` | `none`, `protected_endpoints`, `cli`, `separate_admin_app` |
| `external_api_access` | Third-party access to this system | `none` | `none`, `planned`, `required_at_launch` |

### `api_style` options explained

- **`rest`** — Standard HTTP REST API. Default for most applications.
- **`graphql`** — GraphQL API. Use when clients need flexible querying or the schema is complex and query patterns are hard to predict.
- **`redis_event_layer`** — No traditional HTTP API layer for real-time data. Frontend connects via WebSocket or SSE to a thin gateway that subscribes to Redis channels/streams. Backend services publish events to Redis. HTTP endpoints may still exist for commands/mutations. Best for event-driven, real-time-first applications where the primary data flow is server-to-client push.
- **`hybrid`** — REST for CRUD and commands; Redis event layer for real-time subscriptions. Use when the app has both traditional CRUD and meaningful real-time components.

### `redis_hosting` options explained

- **`self_hosted_ecs`** — Redis container on ECS Fargate. Cheaper, lower latency between services, simpler networking. Fine up to significant scale.
- **`elasticache`** — AWS managed Redis. Higher cost, easier operations at scale, automatic failover.

## Architecture Rules

Non-negotiable unless the project spec explicitly overrides with justification.

1. All clients interact through the same API contract — frontend never accesses the database directly.
2. Prefer simple architecture. Avoid microservices unless explicitly required.
3. Stateful data lives in PostgreSQL. Large files go to S3. Redis is only for cache, queues, pub/sub, and coordination — never primary storage.
4. Background processing uses async workers, not synchronous request handling.
5. Repository is a monorepo unless the spec forbids it.
6. Infrastructure is always defined as code (CDK). No manual AWS console changes.
7. Secrets live in environment variables. No secrets in code or committed config files.
8. Start with a minimal role system (`user`, `read_only`, `sysadmin`). No complex RBAC until there is a real need.

## Repository Structure

```
repo/
  apps/
    web/          # Next.js frontend
    mobile/       # Expo app (phase 2 or later)
  services/
    api/          # FastAPI backend
    workers/      # Background job consumers
  packages/
    shared_types/ # Shared API types and DTOs
    client_sdk/   # Generated or hand-written API client
  infrastructure/
    cdk/          # AWS CDK stacks
  docs/           # AGENTS.md, ARCHITECTURE.md, DEVELOPMENT.md
  tests/
    backend/
    frontend/
  scripts/
  .github/
    workflows/
```

## CI/CD Model

| Trigger | Pipeline |
|---|---|
| Pull Request | lint → typecheck → backend tests → web build |
| Merge to main | deploy dev → run integration tests |
| Production | manual approval gate → deploy production |

---

## Testing Philosophy

Pragmatic. Test critical paths and edge cases. Testing depth increases with system complexity. Don't test framework behaviour.

- Backend: pytest
- Frontend: Playwright for end-to-end

---

## Delivery Phases (default)

| Phase | Scope |
|---|---|
| 1 | Backend API + Web frontend + CI/CD |
| 2 | Redis event system + Background workers |
| 3 | Mobile app |
| 4 | External API access + Scaling |

- [ ] Technology stack confirmed
- [ ] Infrastructure layout defined
- [ ] Repository structure defined
- [ ] Deployment model defined
- [ ] Testing approach defined
- [ ] Phase 1 scope clearly bounded (resist scope creep)
- [ ] Open questions documented for later resolution
